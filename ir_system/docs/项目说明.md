# 大致流程

这个项目现在可以理解成一条“数据采集 -> 数据清洗 -> 建索引 -> 检索 -> 评测 -> 审查 -> 网页展示”的闭环。

## 数据来源
- 从skills爬取详情页
- 解析skill_name / description / weekly_installs / github_stars / repo_url / category
- 做规范化、canonical URL、数值清洗
- 写回 ../data/skills_data.json:1、../data/skills_checkpoint.json:1

## 检索层
检索核心在 src/skills_ir/engine.py:1 和 src/skills_ir/text.py:1。

- tokenize() 做中英文分词和中文切片
- expand_query_tokens() 做 query expansion
- weighted_terms() 按字段权重抽词
- search() 先算 TF-IDF / cosine，再做规则重排
-  _apply_boosts() 叠加 phrase boost、标题命中、描述命中、类别命中、热度权重

## 评测层
离线评测的方法，目前标注的来源为人工标注。评测在 src/skills_ir/evaluation.py:1，评测集在 eval/core_relevance.json。

- 读取评测 query 和 expected_skill_names
- 对每条 query 执行检索
- 看期望 skill 第一次出现在第几名
- 汇总成 Hit@K / MRR@K / Recall@K
- 把失败样例写进 runtime/failure_buckets.json:1
- 把快照写进 runtime/metrics_report.json:1

run-cycle 会自动把这条链跑一遍

## 审查层
你前面提到的“最后一个检查合理性的 agent”，现在主要落在 src/skills_ir/review.py:1 里。

它分两层：

- review_runtime_snapshot()：检查离线评测快照是否合理，比如是不是 benchmark 太小、是不是指
    标被打满
- review_query_results()：检查当前 query 的结果是否合理，比如：
- top 结果里有没有重复 skill
- top1 和 top2 分差是不是太小
- snippet 证据是不是太弱
- 如果 query 本身在 benchmark 里，Top1 是否和期望一致

CLI 的 review-system 就是把这层能力产品化了。

## 网页层
网页在 src/skills_ir/web.py:1 和 src/skills_ir/templates/search.html:1。

现在首页会同时展示：

- 当前 query 的检索结果
- 这个 query 是否命中评测集
- 离线 snapshot 指标
- 实时 benchmark 指标
- query 级合理性审查结果

也就是说，网页不再只是“搜一下然后把结果列出来”，而是把检索、评测、审查三层都挂上去了。

6. 自治闭环
自治主流程在 src/skills_ir/cycle.py:1。

run-cycle 的顺序基本是：

1. 看数据集是否过期
2. 需要的话跑抓取
3. 加载或重建索引
4. 跑数据健康检查
5. 跑评测
6. 生成 cycle report / project state
7. 给出下一轮建议

所以它是整个项目的“总调度入口”。
